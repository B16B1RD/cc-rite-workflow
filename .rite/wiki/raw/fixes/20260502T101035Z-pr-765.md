### Fix Patterns

**CRITICAL: 3-site bang-backtick adjacency self-violation (Asymmetric Fix Transcription の inverse failure)**

Issue #691 が予防対象とするまさに同じ bang-backtick adjacency pattern (`!` + backtick 隣接が parser でコマンド実行をトリガー) を、対策コード (DRIFT-CHECK ANCHOR で symmetric 維持された 2 site + ANCHOR 射程外の 1 site = 計 3 site) が同形に混入していた meta-self-inconsistency を観測。

具体的 root cause: bash の double-quoted string 内に `` `if ! cmd; then` `` (literal backtick pair) を書くと、bash は `if ! cmd; then` を command substitution `$( )` 相当として subshell 実行を試み、incomplete syntax で "syntax error: unexpected end of file" → 該当部分が空文字に置換される (Style B 修正例の中核が silent に消失する UX regression)。

修正パターン: backtick 囲み → single-quote 囲みに置換。3 site 同期適用 (DRIFT-CHECK ANCHOR 射程外の hook script も含む)。

**HIGH: Documentation-implementation drift (sentinel format mismatch)**

`BANG_BACKTICK_CHECK_INVOCATION_FAILED=1` sentinel が Phase 5.4.4.1 grep canonical token (`[CONTEXT] WORKFLOW_INCIDENT=1; type=...; iteration_id=...`) と format 乖離していた。doc は「Phase 5.4.4.1 で auto-track」と主張していたが実装は別経路 (missing-result-pattern fallback / skill_load_failure post-condition check) に流れる。doc 訂正で実装と整合させた (option A)。

### Lessons Learned

1. **DRIFT-CHECK ANCHOR の inverse failure**: ANCHOR は asymmetric な fix transcription (片方 site だけ修正) を防ぐが、両 site に同形 defect を初期投入してしまうケースは ANCHOR では防げない。3 site のうち 1 site (hook script) は ANCHOR 射程外で、これも同形混入していた点が学び。
2. **Self-referential anti-pattern**: 本 PR の予防対象 (bang-backtick adjacency) を対策コード自身が再現する meta-self-inconsistency は、初期 review で全レビュアーが見落とした (5 reviewer 並列でも初期 commit は通過した)。Mutation testing による事後検出が canonical。
3. **Follow-up candidate**: bang-backtick-check.sh に `--print-action-hint` flag を導入し、Style A/B サジェスト文言を 1 source of truth に集約することで 3 site drift を構造的に予防する案が浮上。次 Issue で検討推奨。
