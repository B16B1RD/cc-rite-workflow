### Fix Patterns

1. **Silent legacy fallback observability** (HIGH x2): `2>/dev/null || FALLBACK="<legacy>"` 形式の resolver 失敗 silent fallback を、RITE_DEBUG 経路で `.rite-flow-debug.log` に WARNING 出力する形に修正。pre-tool-bash-guard.sh / post-tool-wm-sync.sh に対称適用 (Wiki: Asymmetric Fix Transcription)。

2. **Schema-2 fast-path API contract code-level enforcement** (HIGH): `check_session_ownership` の schema-2 fast-path に「hook_sid が非空のとき filename SID と一致を verify する」defense-in-depth を追加。将来 resolver を経由しない caller の silent ownership bypass を構造的に防ぐ。

3. **AC-4 contract direct verification** (HIGH): 旧 TC-004 (mismatched SIDs ignored) を 3 ケースに分割: (a) matching SID で 'own' (b) foreign per-session で 'other' (c) 空 hook SID で 'own' (backward-compat)。helper-level の固定により hook integration まで踏み込まずに contract を保護。

### Lessons

- 対称適用 (sibling Issue #680 の lifecycle hooks との対称) を実施したつもりでも、各 caller (Pattern 5 Mode B / post-hook ownership) の context 差異 (resolver 失敗時の影響範囲) を見落とすと defense-in-depth が薄くなる。観測性向上を caller ごとに再検討する必要がある。
- "structural ownership guarantee" を信じる API contract は code-level で enforcement されないと、将来の caller 拡張で silent break する。fast-path に明示的な defense-in-depth check を入れることで invariant を保護。
- helper-level test は hook integration test の代替として有効: 「resolver の path 構築」と「ownership fast-path」のいずれかを変更しても test が落ちる構造を 1 TC で固定できる。
