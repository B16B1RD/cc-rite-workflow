### Fix Patterns
- **success path noise reduction は "truncate" ではなく "selective surface" で解く**: cycle 2 で `clear_git_err` を使って success path の stderr を無条件 truncate した結果、informational (git の `-q` / `--quiet` で消せる) と warning (消せない、debug 価値あり) を区別せず両方捨てる silent regression を生んだ。cycle 3 では `surface_git_warnings` helper で `head -n 10 | grep -iE '^(warning|hint|error):'` により真の警告行のみ filter surface し、informational noise は git 側で既に抑制済みのため silent のまま通す設計に変更。
- **`if ! cmd` は exit code の 3 値分離を破壊する**: bash の `!` 演算子が exit code を binary に畳み込むため、`git diff --quiet` のような rc=0/1/>1 を使うコマンドでは `set +e; cmd; rc=$?; set -e; case "$rc" in 0) ... 1) ... *) error ;; esac` の明示 3 値分離が必須。`if ! ... ; then` で wrap すると real error が silent に正常ケースに丸まる silent regression を生む。
- **tempfile lifecycle は「trap 登録 → mktemp」の順序を全経路で守る**: `ref_err` のような主 cleanup_body trap の外で作成される tempfile は scope 限定 mini-trap (`trap 'rm -f "${ref_err:-}"' EXIT INT TERM HUP`) で mktemp 前から保護し、success 経路の exit 前に明示 `trap - EXIT INT TERM HUP` で disarm する pattern が安全。

### Key learnings (project-specific)
- **noise reduction の設計ルール**: success path で stderr を触るときは「情報量を減らす」のではなく「ノイズと警告を分離する」ことを目的にする。git の `-q` / `--quiet` は前者 (informational の抑制) の責務、helper は後者 (warning の selective surface) の責務、と責務分離する。
- **bash `!` 演算子の罠**: `if ! cmd` は 3 値 exit code を持つコマンドでは使わない。特に `git diff --quiet` / `grep` / `diff` など rc=0/1/>1 の語彙を持つコマンドでは explicit case 文が必須。本 PR cycle 3 MEDIUM #2 が実例。
