# PR #548 Fix Cycle 3 Raw Source

## Context
PR #548 (Issue #547 worktree-based wiki ingest redesign) review-fix loop, cycle 3.

## Findings Resolved (2)

### F-01 (HIGH): wiki/init.md Phase 3.5 setup_rc bash gotcha
- **Pattern**: `if ! cmd; then rc=$?` — bash 仕様上 `$?` が `!` の終了 status (= 0) を捕捉
- **Real RC lost**: setup.sh の rc 1=env error / 2=disabled / 3=worktree add 失敗 が常に 0 と表示
- **User impact**: 原因切り分け不能
- **Fix**: `set +e; cmd; rc=$?; set -e` パターンに置換 (ingest.md Phase 1.3 と対称)
- **Reviewer triple cross-validation**: prompt-engineer + code-quality + error-handling の 3 人が同箇所を独立検出 → boost to HIGH
- **Detection commands**:
  - `bash -c 'if ! (exit 5); then rc=$?; echo $rc; fi'` → `0` (実証)
  - `grep -nE 'if ! .*; then$' --include='*.md' -r .` (anti-pattern grep)

### F-02 (HIGH): pr/cleanup.md Phase 2.6 init.md Phase 番号 drift
- **Drift content**: "init.md Phase 3.5" と記述、実際は Phase 3.5.1 (.gitkeep migration) で worktree-commit.sh を呼ぶ
- **Phase 3.5**: worktree-setup.sh の呼び出しのみ (commit.sh 呼ばない)
- **Cause**: cycle 2 fix で worktree-commit.sh への言及を追加した際に Phase 番号を wrong subsection に当ててしまった

## Common Pattern: Asymmetric Fix Transcription

両 finding とも cycle 2 で行った修正の対称箇所への伝播漏れ。

### Pattern definition
ある fix を 1 箇所に適用したとき、同じ pattern を持つ「対称箇所」に同じ fix を伝播させ忘れる failure mode。
特に以下のケースで発生しやすい:
1. **同型 bash idiom が複数 phase に存在**: ingest.md Phase 1.3 / init.md Phase 3.5 のように同一 pattern が異なる場所にある
2. **invocation chain や cross-reference の Phase 番号書き換え**: 1 箇所を直すと別 doc の参照番号も連動して直す必要があるが見落としがち

### Detection heuristic
fix 直後に必ず実行する: `grep -rn "{anti-pattern}" --include='*.md'` で同一 anti-pattern の残存をスキャン。
同じく fix で Phase 番号を変えたら `grep -rn "Phase {old_number}" --include='*.md'` で参照漏れをチェック。
