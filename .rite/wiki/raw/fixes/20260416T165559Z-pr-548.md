### Fix Patterns

PR #548 (Issue #547) のレビュー指摘 21 件を 1 サイクルで全て解決した経験から得られた知見:

1. **plugin_root 解決の前置原則**: bash block 冒頭で `$plugin_root` を参照する場合、その変数を出力する Phase は **必ず先行する Phase に配置する**。後続 Phase で初めて解決される変数を空展開で参照させると ENOENT 系の silent 失敗が発生し、ingest 機能全体が常に「worktree_setup_failed」で早期 return する致命的な broken 状態になる。今回は Phase 1.3 → Phase 1.2 への前置で解消。
2. **plugin_root inline 解決の境界**: command file が複数 Phase で plugin_root を使う場合は専用 Phase を立てる。1 箇所だけで使う場合は inline 解決 (Phase 8.3 冒頭等) に閉じる方が drift しにくい。lint.md は後者を採用。
3. **`drift_count` のような pipeline 由来の数値は必ず default + regex バリデーション**: `wc -l | tr -d` の出力が空文字になりうる経路 (find が permission denied で 0 件返す等) では `[ -gt 0 ]` 比較が「空文字 < 1」として silent pass する。`${var:-0}` + `[[ =~ ^[0-9]+$ ]]` の二段ガードを必ず添える。
4. **3 変数 nest vs has_changes 単一変数**: 同種スクリプトの既存パターン (wiki-ingest-commit.sh has_changes) との対称性を保つ場合、early-exit の集約変数だけ追加して granular vars は diagnostic 出力 (DRY_RUN 等) のために保持するのが両立解。
5. **trap action リセットの最小化**: `trap - EXIT INT TERM HUP` は将来追加される他 trap も無効化する副作用を持つ。tempfile 削除直後に `var=""` で空文字代入して scope を絞ってから trap 解除すれば、後続の trap 追加に対して安全。
6. **`exec 9>...` の hard fail 対策**: `set -euo pipefail` 配下では `exec` の I/O 失敗 (permission denied / read-only fs / disk full) も script 全体を hard fail させる。advisory lock が best-effort であるべき場面では subshell `( exec 9>... ) 2>/dev/null` で test してから本当の `exec` に進む 2 段階パターンが安全。
7. **branch name の path traversal hardening**: git ref name validation regex は `[A-Za-z0-9._/-]+$` だけだと `..` や `//` を許容してしまう。`git check-ref-format` 準拠の subset として `..`/`//`/先頭`.` の 3 パターンを明示拒否する。
8. **mktemp 失敗の silent 握り潰し禁止**: `mktemp ... || echo ""` パターンは disk full / inode 枯渇を silent に握り潰し、後続の stderr capture が失われる。`if ! var=$(mktemp ...); then WARNING; var=""; fi` で WARNING を可視化する。
9. **wiki worktree の永続化原則**: `.rite/wiki-worktree/` は cleanup で削除しない設計 (Issue #547)。理由を `.gitignore` の inline コメントと cleanup.md Phase 2.6 に明記しないと将来の reviewer/contributor が削除提案で無効化してしまう。
