### Fix Patterns

Shared lib の contract 違反 findings を解決する過程で得られた経験則:

1. **Lib helper の contract は実装と同じ rigour で書く**: `caller owns shell options` /
   `caller's outer trap is restored` という header 宣言に対して実装が `set -e` 強制 /
   `trap - EXIT ...` 消去を行うと、将来 caller が function 前に trap を設置した瞬間に
   silent regression になる。`trap -p` + `eval` で outer trap を保存・復元し、errexit
   も `$-` で状態保存 + 条件復元する pattern が canonical。

2. **`cd "$repo_root"` 後の `$(dirname "$0")` は不安定**: 相対 path invocation
   (`bash ./scripts/foo.sh`) で cd 後に `$(dirname "$0")` は元の相対 string の dirname を
   そのまま返し `./scripts` として解釈され、`$repo_root/./scripts/lib/...` を探索して source
   失敗する。`_SCRIPT_DIR="$(cd -P "$(dirname "${BASH_SOURCE[0]}")" && pwd)"` を cd 前に
   実行し、絶対 path に canonicalize するのが canonical convention (既存 hook scripts も同じ)。

3. **stderr tempfile を per-step に分離**: add/diff/commit/push の 4 step で 1 つの tempfile
   を流用すると、後段 step 失敗時に前段の warning が混在表示されて root cause 診断が困難になる。
   diff_err など dedicated tempfile を確保する。

4. **silent fallback (`|| echo "HEAD"`) は push 失敗誤診の温床**: detached HEAD / corrupt
   worktree で `rev-parse --abbrev-ref HEAD` が失敗したとき literal "HEAD" を push target
   にするのは push 権限エラーとして誤診される経路を生む。caller から検証済み branch を明示
   引数で渡す signature の方が robust。
