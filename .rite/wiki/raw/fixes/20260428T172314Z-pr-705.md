### Fix Patterns
- **Whitelist broken refs**: SoT 自身の Whitelist テーブルが「撤去済みファイル」「存在しない skill」「canonical 文書ではない出典」を指す broken reference 3 件。修正方針は (a) 撤去済み: Whitelist から削除し Note で歴史的経緯のみ残す (b) 存在しない skill: 実体が存在する canonical 文書に出典変更 (c) canonical でない出典: 真の canonical 文書 (`bottleneck-detection.md` 等) を採用
- **PR cross-reference fragility**: 並行する未マージ PR (#688) を「実プロジェクト由来」と称して引用すると、本 PR マージ時点で「実プロジェクト」が空になる。対処は「未マージ branch である事実を脚注で明示し、引用の意図を Bad 例として独立化」
- **identifier 表記揺れ**: 同一文書内で原則 ID を short form (`no_line_reference`) と canonical form (`no_line_or_cycle_reference`) で混在させると、後続 Issue で grep anchor として使う際の drift 源になる。SoT を作る際は「原則一覧テーブル」の表記を canonical として全文中で統一する
- **regex character class incompleteness**: shell プロジェクトはハイフン含み file 名が一般的 (`_emit-cross-session-incident.sh`)。ファイル名 regex で `[a-z_]` のみを使うと false negative になる。`[a-zA-Z0-9_./-]` 等で hyphen + Capital + 数字 + path separator を許容する

### 経験則 (Experiential Knowledge)
新規 SoT を作成する際、文書内部から他リポジトリ要素への path 参照を含める場合、参照先の存在を **本 PR がマージされる時点の origin/develop** で機械的に検証する。並行 PR / 撤去済み artifact / canonical でない参照は SoT を最初の bad example にする。レビュー時は SoT のすべての path 参照を `git ls-tree origin/develop` で existence check する。
