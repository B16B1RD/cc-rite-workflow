## 経験則 (Wiki 蓄積候補)

### E-1: Factual regression in status-label annotation
Status labels (OPEN/CLOSED) を手書きで追加する際、gh API 実測で verify せずに commit すると cycle 2 型 factual regression を再生産する。cycle 5 F-C4-06 が「OPEN/CLOSED 追加で読者に正確に伝える」目的だったのに、追加した label 自体が factually 誤りだった皮肉。将来の mitigation: `gh issue view N --json state,closedAt` で機械的に verify する script 化、または anti-pattern.md から status label 自体を削除して gh dynamic fetch に任せる構造変更。

### E-2: Protection theater in test pin claims
test コメントが「N site を pin」と主張しても実 assert が 1 site のみの場合、regression 検出インフラへの信頼を破壊する (F-C6-03)。test reviewer が mutation test で実証する方法論 (`sed` で canonical phrase を 1 文字 drift させて test suite を再実行) が drift-check integrity を empirical に検証する pattern。

### E-3: Canonical phrase unification は arm-wide で適用すべき
cycle 4/5 で cleanup arm を 3 site byte-identical にしたが ingest arm を symmetrize しなかったため cycle 6 で F-C6-02 として再浮上。「同型の issue を片側のみ修正する」と drift が凍結される。defense-in-depth / terminal 規約系の lexical consistency は常に arm 全体 (cleanup 系 + ingest 系) を scope に入れる必要がある。

### E-4: Markdown channel separation モデルは前後空行要求の両方を扱うべき
cycle 5 の markdown channel separation 説明は CommonMark HTML block type 2 の **後方空行要求** の吸収のみを扱うが、cleanup 旧仕様 (#652 Root Cause) の可視化原因は **前方空行要求** (list item → HTML block 境界)。完全な因果 chain は「前方 (list item→HTML block で要求あり) + 後方 (markdown 末尾で吸収)」の双方を論じる必要がある (F-C6-15 SUGGESTION)。
