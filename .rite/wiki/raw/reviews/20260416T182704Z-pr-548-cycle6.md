## Key Recurring Pattern — "Asymmetric Fix Transcription"

The dominant recurring finding across cycles 3–5 was not bugs in the code's runtime
behavior but **asymmetric transcription** between paired scripts:
`wiki-ingest-commit.sh` vs `wiki-worktree-commit.sh` vs `wiki-worktree-setup.sh`.

Each fix cycle applied a policy (fail-fast on git rev-parse error / full stderr surface
via tempfile / `set +e; rc=$?; set -e` explicit rc check) to one or two of the three
scripts but not the third, triggering cross-validation findings from at least two
reviewers in the next cycle. Cycle 5 finally closed the asymmetry by:

1. **F-01 (HIGH)**: Replaced silent `2>/dev/null` rev-parse with full capture/surface
   pattern (tempfile + head -3 + rc check) matching the companion script.
2. **F-02 (LOW)**: Mechanical 2-space de-indent of lines 353-453 (removed orphan
   4-space indent left behind after deleting an outer `if [ "$wt_head" = "$wiki_branch" ]`).
3. **F-03 (LOW)**: Replaced stale line-number cross-references with relative anchors
   (e.g., "fm_err / ref_err / stage_dir tempfile") that survive future line-number drift.

## Lesson for Future Fix Cycles

When a fix applies to one of a pair (or trio) of paired scripts, the reviewer
should explicitly list all paired files in the finding's
`files_to_propagate_to` metadata, and the fix phase should apply the change
to all of them atomically. Otherwise the next review cycle will surface the
missing counterpart(s) as a "new" finding, inflating review cycles.

Observed in this PR: `propagation_applied: 0` across all 5 fix cycles — no fix
attempted automatic propagation. Each asymmetry was caught manually by
cross-validation between reviewers, at the cost of 3 extra cycles.
