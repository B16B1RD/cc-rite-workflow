### Key learnings (project-specific experiential knowledge)
- **stderr noise reduction の設計原則**: success path で stderr を無条件 truncate するのは anti-pattern。正しい方法は (a) `grep` で warning/hint 行のみ selective surface、または (b) `git <cmd> -q` + stderr → bit bucket で informational のみ抑制し真の error は残す設計。warning を silent drop すると「修正不要な情報過多」と「本当の warning の silent loss」を同じ解決策で扱うことになり後者の fail mode を増やす。
- **`if ! cmd` パターンは exit code の 3 値分離を破壊する**: bash の `!` 演算子は exit code を binary (0 / non-zero) に畳み込むため、`git diff --quiet` のような rc=0 (clean) / rc=1 (has-diff, 正常) / rc>1 (real error) の 3 値を使うコマンドでは `case` 文 + 明示 exit code 判定が必須。`if ! ... ; then` で wrap すると real error が silent に正常ケースに丸まる。
- **cleanup 関数に統合される tempfile のライフサイクル管理**: `git_err` のような tempfile を cleanup_body 関数内で削除する設計は、mktemp が trap 登録 **後** に行われる順序を守れば signal path でも orphan を防ぐ。ただし同じ file type の別変数 (`ref_err` のような script 上部で trap 前に作成される tempfile) は scope 限定 mini-trap を別途必要とする。「この PR で新規導入した tempfile」と「既存の tempfile」の両方を漏れなく cleanup 経路に含めること。
